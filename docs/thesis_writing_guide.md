## 4. 写作提纲（按“你现在就能写”的粒度）

### 4.1 绪论怎么写
- 背景：程序设计课程强调实践，但传统平台对个体差异支持不足。
- 问题：只给题不指导、过程数据利用率低、推荐难解释。
- 目标：构建“可追踪、可解释、可验证”的教学支持系统。
- 贡献：
  - 多模块一体化实现（题库、评测、画像、推荐、导出）；
  - 推荐依据可视化展示；
  - 引入推荐效果追踪与保守准确度判断；
  - 使用 AST 做结构化诊断辅助。

### 4.2 系统设计章节怎么写
- 先给总体架构图（浏览器、Flask、SQLite）。
- 再按数据流写：
  1) 学生登录；
  2) 做题提交；
  3) 生成学习日志；
  4) 聚合指标形成画像；
  5) 触发推荐；
  6) 收集反馈并评估效果；
  7) 更新推荐准确度。
- 每个模块要有“输入-处理-输出”三段式描述。

### 4.3 实现章节怎么写
- 以核心函数为主线写实现：
  - `build_user_metrics`：构建知识点维度指标；
  - `ensure_recommendation_for_user`：推荐生成；
  - `evaluate_recommendation_effect`：推荐后效果评估；
  - `calculate_recommendation_accuracy`：准确度标签判定；
  - `/api/submit`：提交评测 + 日志落库；
  - `/dashboard`：推荐与画像综合展示。
- 每小节给 1 段“业务意义”+ 1 段“实现逻辑”+ 1 张“运行截图/表格”。

### 4.4 测试与分析章节怎么写
- 功能测试：注册、登录、筛选、组卷、提交、推荐、反馈、导出。
- 数据分析：
  - 推荐前后正确率变化；
  - 推荐前后平均用时变化；
  - 不同证据等级下推荐准确度分布；
  - 学生主观反馈与系统判断一致性。
- 案例分析：选 1~2 个学生完整学习轨迹做“前后对比”。

- 整体正确率、平均完成时长、累计日志数；

- 推荐后同类题正确率变化（核心）；
- 推荐后同类题用时变化（核心）。

### 5.4 可直接在代码中提炼的“创新点表达”
- 推荐不仅给结果，还给“依据映射 + 证据等级 + 诊断摘要”；
- 推荐准确度采用“客观数据 + 学生主观反馈”联合评估；
- 当样本不足时使用“证据不足/待继续观察”保守标签，避免过度结论；
- 通过学习日志字段扩展实现从“单次评测”到“连续学习分析”的演进。

## 9. 逐文件深读结果（可直接写进“系统实现”）

> 说明：以下内容按你当前仓库中的实际文件组织整理，便于你把“代码实现”一节写成“模块实现说明”。

### 9.1 后端主程序 `app.py`

#### A. 基础配置与常量
- 系统采用 Flask 单体结构，数据库为 SQLite（`system.db`）。
- 定义了章节到知识点的映射（`SKILL_MAP`），用于把题目 tag 统一转换为能力维度。
- 定义了推荐反馈评分映射（匹配度、帮助度），用于主观反馈量化。

#### B. 数据库初始化与演进
- 创建四张核心表：`problems`、`learning_logs`、`users`、`recommendation_events`。
- 使用 `ensure_columns` 机制实现“增量字段补齐”，这点可写成“系统可演进性设计”。
- `learning_logs` 与 `recommendation_events` 已包含推荐闭环相关字段，支持论文中的过程跟踪分析。

#### C. 评测与诊断逻辑
- `analyze_code_structure`：用 AST 提取变量数、循环数、递归特征。
- `calculate_pass_metrics`：以规则方式模拟通过率、运行时间、内存占用。
- `classify_error_type`：按语法、输出、分支、递归边界等分类错误类型。

#### D. 学习画像与推荐逻辑
- `build_user_metrics`：按知识点汇总尝试次数、正确率、平均时长、错误类型。
- `ensure_recommendation_for_user`：满足最小样本后，对薄弱项发起推荐。
- `compute_recommendation_scores`：计算命中度、难度匹配度、依据可信度、证据等级。
- `derive_rule_adjustment`：根据历史推荐效果动态调节难度。

#### E. 推荐效果评估与准确度判断
- `evaluate_recommendation_effect`：计算推荐后同类题表现（正确率、用时、错误分布）。
- `calculate_recommendation_accuracy`：融合客观效果 + 主观反馈得出准确度分数与标签。
- 当样本不足时返回“证据不足/待继续观察”，体现保守判断策略。

#### F. 路由与页面数据组织
- `/`、`/paper`：题库展示与组卷。
- `/api/submit`：提交代码、写日志、触发推荐评估。
- `/api/recommendation_feedback`：采集学生反馈并刷新准确度。
- `/dashboard`：聚合画像、推荐事件与追踪统计。
- `/export_data`：导出可用于论文统计分析的 CSV。

### 9.2 前端模板（论文中“系统界面实现”）
- `templates/index.html`：题库主页，支持章节/难度筛选和组卷入口。
- `templates/solve.html`：做题与提交页，展示推荐依据、证据等级、反馈交互、AST 报告。
- `templates/dashboard.html`：学习画像总览、推荐摘要、推荐追踪、日志表、雷达图。
- `templates/login.html`/`register.html`：认证入口。
- `templates/leaderboard.html`：排行榜模块。

### 9.3 数据与文档文件
- `mock_data.py`：可用于快速生成日志样本，便于论文初期截图与演示。
- `README.md`：项目亮点与界面说明，可转化为绪论和系统概述草稿。

## 10. 完整写作流程（从“今天开始”到“提交终稿”）

### 阶段 0：定题与边界（半天）
1. 固定题目（不要再频繁改）。
2. 固定研究边界：本系统聚焦“程序设计课程教学支持”，不扩展到通用 OJ 或商业推荐。
3. 固定论文主线：
   - 学习日志建模；
   - 可解释推荐；
   - 推荐效果验证；
   - 保守准确度结论。

### 阶段 1：素材沉淀（1~2 天）
1. 从系统导出 `data.csv`（用于实验统计）。
2. 至少准备 3 组典型学习案例（高水平/中等/基础薄弱）。
3. 截图固定场景：
   - 首页、登录、注册；
   - 推荐题提交前；
   - 推荐题提交后（有 AST 与结果反馈）；
   - Dashboard（总览、依据映射、追踪记录）；
   - 导出数据页面或导出结果。

### 阶段 2：先写“最容易写”的章节（2~3 天）
1. **第3章需求分析**：按功能模块直接展开。
2. **第4章系统设计**：把数据流与表结构写清楚。
3. **第5章系统实现**：按函数链路写，不要按页面流水账写。

### 阶段 3：实验与结果（2 天）
1. 定义指标：正确率、用时、推荐准确度、证据等级分布。
2. 对比维度：推荐前 vs 推荐后；有反馈 vs 无反馈；样本充足 vs 样本不足。
3. 输出图表：折线图、柱状图、表格对比。
4. 形成 2 个完整案例：
   - 案例 A：推荐有效（准确度提升）；
   - 案例 B：证据不足（系统保守判断）。

### 阶段 4：补绪论与总结（1 天）
1. 绪论最后写，避免前后不一致。
2. 总结要量化：做了什么、效果如何、局限在哪里。

### 阶段 5：格式与查重前处理（1 天）
1. 统一图表编号、章节编号、参考文献格式。
2. 统一术语：推荐准确度、证据等级、验证状态等关键词不要混用。
3. 检查每个结论是否有图/表/数据支撑。

## 14. 最后给你的写作原则（很重要）
1. 论文不是“功能说明书”，必须有“分析与比较”。
2. 结论必须数据化，不能只写“效果较好”。
3. 每个创新点都要有对应实现与图表支撑。
4. 样本不足要主动承认并给出改进方向，这不是减分，反而体现严谨。
